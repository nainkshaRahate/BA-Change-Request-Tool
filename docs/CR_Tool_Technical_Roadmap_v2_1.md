## Section 1 — The Official CR Process (Ground Truth)

This section captures the process exactly as defined in the CR Process Guide V1. The app must enforce all of these rules.

### 1.1 Weekly Cycle & Deadlines

| Day | Deadline | Action | Owner |
|---|---|---|---|
| Wednesday | COB | CR submitted to SharePoint folder and entered in app | CR Owner |
| Thursday/Friday | — | CR Programme Meeting — CR Owner presents each CR | CR Owner or delegate |
| Friday | COB | All post-meeting edits completed in SharePoint and app | CR Owner / PMO |
| Monday | Morning | PMO sends final CRs to SG attendees for review | PMO |
| Monday | By end of day | SG pack (4×4 + status reports for Red/Amber) updated | PMO |
| Tuesday | — | AIPSG / SG Meeting — Decisions made | Steering Group |

The app must surface these deadlines visually. Any CR with a Wednesday submission deadline that hasn't been submitted by Wednesday COB should show as overdue in the PMO dashboard.

### 1.2 The Full 10-Step Lifecycle

| Step | Name | Owner | Key Rule |
|---|---|---|---|
| 0 | Draft / Pipeline | CR Owner | CR exists in app but not formally submitted. No deadlines apply yet. |
| 1 | Submission | CR Owner | Upload by COB Wednesday. Must use correct naming convention. |
| 2 | PMO Review | PMO | Validate completeness and template compliance. Accept or return. |
| 3 | Assessment Distribution | PMO | Send to all 6 stakeholder groups. RAG defaults to Pale Blue. |
| 4 | Stakeholder Assessment | Each stakeholder | Provide RAG (G/A/R/Grey), Impact Statement, Reviewer name and date. No Pale Blue allowed at completion. |
| 5 | PMO Consolidation | PMO | Compile all assessments, calculate cost impact, resolve Red/Amber. |
| 6 | Programme Meeting | CR Owner or delegate | Present CR. If no attendee, CR is automatically deferred — not discussed. |
| 7 | Pre-SG Distribution | PMO | Send to SG attendees by Monday morning. |
| 8 | SG Meeting | SG Members | Decision: Approved / Deferred / Rejected. Held on Tuesday. |
| 9 | Post-Meeting Edits | PMO | Complete all edits by COB Friday. Update tracker and version. |
| 10 | Implementation & Closure | PMO | Notify teams, track progress, verify, close CR. |

### 1.3 CR Naming Convention (Exact Rules from Guide V1)

Two separate naming formats are required:

**File name (for SharePoint document library):**
```
CR<number> - <Programme name> – <Title>
Example: CR123 - A380 Vector - Upper Desk Cabin Bed
Single digit: CR05 - A380 Vector - Upper Desk Cabin Bed
```

**CR Title (displayed in the app and on the slide):**
```
<Programme Code>: <CR Number> - <Title>  (title in UPPER CASE)
Example: A380: CR123 - UPPER DESK CABIN BED
Single digit: A380: CR05 - UPPER DESK CABIN BED
```

The app auto-formats the CR Title display from the three fields (Programme Code, CR Number, Title). The file name is generated by the "Generate Slide" Power Automate flow.

### 1.4 The RAG Status System (Complete Definition)

Five possible values. This is the exact set from the CR Process Guide:

| Colour | Meaning | When to use | Hex code for app |
|---|---|---|---|
| **Green** | Acceptable to approving area | Stakeholder reviewed — no concerns | `#27AE60` |
| **Amber** | Adds risk to approving area | Stakeholder reviewed — has concerns | `#F39C12` |
| **Red** | Unacceptable impact to approving area | Stakeholder reviewed — blocking | `#C0392B` |
| **Grey** | No additional stakeholder | Used for the OTHER row only, when that stakeholder category is not relevant to this CR | `#95A5A6` |
| **Pale Blue** | Stakeholder yet to review | **The default state.** Set automatically when CR is distributed. Must be cleared before AIPSG Ready. | `#AED6F1` |

**The critical rule:** No CR may be marked "AIPSG Ready" if any stakeholder row still shows Pale Blue. The app must enforce this with a hard validation block.

**Grey vs Pale Blue distinction:** Grey means "this stakeholder is not applicable to this CR." Pale Blue means "this stakeholder is applicable but has not yet responded." These are completely different situations and must not be confused.

Create a dedicated SharePoint site named: **`BA-CR-Tool`** (or align with existing BA SharePoint naming conventions).

### Screen 2: View Change Requests (`scrViewCRs`)

- Programme dropdown (All + distinct ProgrammeName values from list)
- AIPSG Date dropdown (bound to AIPSGMeetings list, active meetings only)

**Gallery Items property:**
```powerfx
Sort(
    Filter(
        CRRequests,
        // Search box:
        IsBlank(txtSearch.Text) Or
            txtSearch.Text in ThisRecord.Title Or
            txtSearch.Text in ThisRecord.CRNumber,
        // Status filter:
        ddlStatus.Selected.Value = "All" Or CRStatus = ddlStatus.Selected.Value,

    AIPSGTargetDate,
    Ascending
)
```

**Each gallery row shows:**

- Status pill (colour from table in Section 5.4 below)
- AIPSG Target Date (amber if ≤14 days, red if overdue, normal if fine)
- Programme name

```powerfx
If(
```

**Submit Formally validation — before patching:**
```powerfx
// Submit button OnSelect:
If(
    IsBlank(txtCRNumber.Text) Or
    Not(Left(txtCRNumber.Text, 2) = "CR") Or
    IsBlank(txtProgrammeCode.Text) Or
    IsBlank(txtProgrammeName.Text) Or
    IsBlank(txtCRTitlePart.Text) Or
    IsBlank(pplProjectManager.Selected) Or
    IsBlank(pplSponsor.Selected) Or
    IsBlank(ddlAIPSG.Selected) Or
    IsBlank(taCRDescription.Text),
    // Show validation error:
    Set(gblValidationMsg, "Please complete all fields marked *"),
    // Check for duplicate CR Number:
    CountRows(Filter(CRRequests, CRNumber = txtCRNumber.Text)) > 0,
    Set(gblValidationMsg, "CR Number " & txtCRNumber.Text & " already exists. Please check."),
    // All valid — patch:
    Set(
        gblSavedCR,
        Patch(
            CRRequests,
            If(gblCRMode = "New", Defaults(CRRequests), galCRList.Selected),
            {
                Title: Upper(txtProgrammeCode.Text) & ": " & txtCRNumber.Text & " - " & Upper(txtCRTitlePart.Text),
                CRNumber: txtCRNumber.Text,
                ProgrammeCode: Upper(txtProgrammeCode.Text),
                ProgrammeName: txtProgrammeName.Text,
                CRTitle: Upper(txtCRTitlePart.Text),
                Supplier: txtSupplier.Text,
                SupplierReference: txtSupplierRef.Text,
                ProjectManager: pplProjectManager.Selected,
                CRSponsor: pplSponsor.Selected,
                AIPSGTargetDate: ddlAIPSG.Selected.MeetingDate,
                SubmissionDeadline: ddlAIPSG.Selected.SubmissionDeadline,
                CRDescription: taCRDescription.Text,
                ReasonForChange: taReason.Text,
                DoNothingOption: taDoNothing.Text,
                AdditionalInformation: taAdditional.Text,
                RE_Delta_PerAircraft: txtRE.Text,
                NRE_Delta: txtNRE.Text,
                TotalCostDelta: txtTotal.Text,
                WeightDelta_PerAircraft: txtW1.Text,
                WeightDelta_AllAircraft: txtWAll.Text,
                AnnualCoWImpact: txtCoW.Text,
                CostCurrency: ddlCurrency.Selected.Value,
                IsConfidential: ddlConfidentiality.Selected.Value = "Confidential",
                IsDraft: false,
                CRStatus: "Submitted",
                Originator: {Claims: "i:0#.f|membership|" & User().Email, DisplayName: User().FullName},
                SubmittedDate: Today()
            }
        )
    );
    'PowerAutomate-OnCRSubmit'.Run(gblSavedCR.CRNumber);
    Navigate(scrViewCRs, ScreenTransition.None);
    Notify(txtCRNumber.Text & " submitted successfully.", NotificationType.Success)
)
```

## Section 6 — Power Automate Flows (4 Flows)

### Flow 1: `CR-OnSubmit-NotifyPMO`
**Trigger:** New item in `CRRequests` where `IsDraft = false`
**Actions:**
1. Send email to PMO: "New CR submitted — {Title} by {Originator}. Please review and distribute."
2. Include a direct link to the app with the CR pre-selected.

---

### Flow 2: `CR-OnDistribute-CreateStakeholderRows`
**Trigger:** Item updated in `CRRequests` where `CRStatus` changes to `"In Assessment"`
**Actions:**
1. Check if stakeholder rows already exist for this CR (idempotent — prevents duplicates if flow runs twice).
2. If not, loop through the 6 group names and create one row in `CRStakeholderReviews` per group:
   - RAGStatus = "Pale Blue" (the official default per CR Process Guide V1)
   - IsComplete = false
3. Send email to each stakeholder group's distribution address: "CR {CRNumber} — {CRTitle} requires your assessment. Please log in to the app and submit your RAG status and impact statement by {SubmissionDeadline}."

**Hard-coded stakeholder group list for the loop:**
```json
["Engineering Programmes", "Engineering Technical", "Onboard Experience", "Finance", "Procurement", "Other"]
```

---

### Flow 3: `CR-OnPreSGDistribution-EmailSGAttendees`
**Trigger:** Item updated in `CRRequests` where `CRStatus` changes to `"Pre-SG Distribution"`
**Actions:**
1. Get the SG attendee list from `CRUserAdmin` where Role contains "Steering Group" (or use a fixed distribution list).
2. Send email: "SG Pack — {AIPSGTargetDate} AIPSG Meeting. Please find attached the CR for review: {Title}. The SG Meeting is on Tuesday."
3. Include a link to the app CR Detail screen.

---

### Flow 4: `CR-WeeklyReminder-Monday`
**Trigger:** Scheduled — every Monday at 7:30 AM
**Actions:**
1. Get all CRs where CRStatus is "In Assessment" or "PMO Consolidation" and AIPSGTargetDate is within 7 days.
2. Count how many still have Pale Blue stakeholder rows.
3. Build an HTML summary table.
4. Send to PMO: "Weekly CR Status Digest — {N} CRs active, {X} with outstanding Pale Blue assessments."

This flow directly replaces the manual weekly PPT update task.

## Section 7 — Governance Rules Enforced in the App

This table maps every rule from the CR Process Guide V1 to the specific app control that enforces it.

| Rule from Guide V1 | Where enforced in app |
|---|---|
| Use correct naming convention: `CR<number> - <Programme> – <Title>` | `scrSubmitCR` — CR Number field validation. Title auto-formatted on save. |
| Single-digit CRs must be zero-padded (CR05 not CR5) | `scrSubmitCR` — Regex-style check: `If(Len(Mid(txtCRNumber.Text, 3)) < 2, show error)` |
| Submit by COB Wednesday | `scrDashboard` — overdue tile. Flow 4 Monday reminder. `scrViewCRs` — red date if past deadline. |
| CR Owner or delegate must attend Programme Meeting | `scrProgrammeMeeting` — "Proceed" button blocked if AttendanceConfirmed = false and DelegateAttending is blank. |
| Post-meeting edits by COB Friday | Not auto-enforced (PMO judgement call) — but `scrPMOConsolidation` shows days since Programme Meeting. |
| Pre-SG Distribution by Monday morning | Flow 3 sends emails immediately when status reaches "Pre-SG Distribution". PMO triggers this manually. |
| No Pale Blue by SG stage | "Mark Ready for SG" button hard-blocked if any Pale Blue rows remain. Warning banner on Part B screen. |
| Grey only for OTHER row with no applicable stakeholder | RAG dropdown for non-Other rows excludes Grey as an option. |
| PMO updates SG Pack by Monday | Flow 4 reminds PMO on Monday morning. "Generate Slide" button produces the updated PPTX. |

## Section 8 — The Generate Slide Feature

This is the highest-value feature in the tool. It eliminates the weekly manual PPT update.

**How it works:**
1. PMO or CR Owner presses "Generate Slide" on `scrCRDetail`.
2. A Power Automate flow retrieves all CR fields and all 6 stakeholder review rows.
3. The flow copies the master template from the `CR-Templates` SharePoint library.
4. It uses the **Office Scripts** action in Power Automate to replace placeholder text in the copied file:
   - `{{CR_NUMBER}}` → CR123
   - `{{PROGRAMME}}` → A380 Horizon
   - `{{CR_TITLE}}` → UPPER DESK CABIN BED
   - `{{SPONSOR}}` → James Dobberson
   - `{{DESCRIPTION}}` → [full description text]
   - `{{ENG_PROG_RAG}}` → Green (which translates to the green cell fill)
   - `{{ENG_PROG_IMPACT}}` → [impact statement text]
   - … and so on for all 6 stakeholder rows
5. The file is saved to the correct monthly folder in `CR-Files` with the correct file name syntax: `CR123 - A380 Vector - Upper Desk Cabin Bed.pptx`
6. The flow returns the SharePoint link. The app displays it as a "Download Slide" button.

**Template preparation (one-time task):**
In the master PowerPoint template, replace every dynamic value with a unique placeholder. Keep the visual design, colours, and table structure exactly as-is. Only the text content changes.

## Summary: What This Tool Replaces, Exactly

| Current manual step | Tool replacement |
|---|---|
| Programme Manager copies the template, saves to SharePoint folder, fills in details | `scrSubmitCR` form — structured fields, auto-formatted title, auto-saves to correct status |
| Nainksha emails the PPT to stakeholder groups | Flow 2 sends targeted emails automatically when PMO accepts the CR |
| Stakeholders email Nainksha their RAG and comments, or edit the shared PPT | Stakeholders log into the app, update their own row in `scrStakeholderAssessment` |
| Nainksha manually updates the PPT consolidation every week | Live data — the app always shows current state. Flow 4 sends a Monday morning digest automatically |
| Nainksha manually checks for Pale Blue before the SG meeting | Hard block in app — "Mark Ready for SG" button refuses to proceed if any Pale Blue remains |
| Nainksha prepares the SG Pack slides manually each Monday | "Generate Slide" button produces a correctly formatted PPTX from live data in one click |
| No way to search or filter CRs across programmes | `scrViewCRs` — searchable, filterable gallery across all programmes and dates |
| No history of decisions or changes | Version History on SharePoint list. Every status change is logged. |
